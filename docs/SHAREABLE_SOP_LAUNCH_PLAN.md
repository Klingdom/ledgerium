## 1. Feature Definition

### What is a Shareable SOP Link?

A shareable SOP link is a permanent, publicly accessible URL pointing to a read-only view of a single Ledgerium workflow's SOP output. The link is generated by the workflow's owner from the workflow report page. No login is required to view it.

The URL format is:

```
https://app.ledgerium.ai/shared/[share_token]
```

Where `share_token` is a randomly generated, non-sequential, non-guessable identifier (UUID v4 or equivalent). Share tokens are not enumerable and do not expose the numeric workflow ID.

### What does the viewer see?

The shared view is a standalone page, separate from the authenticated app shell. It renders one workflow's output in a clean, read-only layout.

**Viewer content — Free tier owner:**

- Workflow name and description
- SOP output: all steps, instructions, timing, and system labels (the full SOP as specified in process-map-sop-standard.md)
- Process map (interactive React Flow visualization, view-only)
- Quality indicators: step count, duration, confidence score, friction count
- Source attribution footer: "Generated by Ledgerium AI from a live browser recording."
- "Made with Ledgerium" banner CTA (described in Section 3)

**What is NOT shown to the viewer:**

- The authenticated dashboard
- Other workflows belonging to the owner
- Raw event data or session bundle contents
- Personal account information
- Agent intelligence output (Pro-gated; described in Section 7)
- The workflow evidence tab (contains raw events; never surfaced in shared view)

### What the owner controls

The owner can:
- Generate a share link (creates or retrieves the existing share token)
- Revoke the share link (deactivates the token; existing URLs return a 404-style "This workflow is no longer shared" page)
- Re-share the same workflow (generates a new token)
- On Pro: enable/disable viewer analytics (view count visible in their dashboard)
- On Pro: enable password protection (viewer must enter a password before the SOP loads)
- On Enterprise: replace "Made with Ledgerium" branding with their own logo

### Data model additions required

The existing `WorkflowShare` schema exists per PRD_v2.md. Confirm or add:

```
WorkflowShare
  id              UUID, PK
  workflow_id     FK → Workflow
  owner_user_id   FK → User
  share_token     UUID, unique, indexed
  is_active       boolean, default true
  password_hash   nullable (Pro feature)
  view_count      integer, default 0
  created_at      timestamp
  updated_at      timestamp
  last_viewed_at  nullable timestamp
```

No sensitive event data is stored in this table. The share token grants read access to the workflow's already-computed SOP, process map, and quality indicators only.

## 2. Viral Loop Mechanics

### The loop

```
Owner records workflow
  → Owner views SOP output in web app
  → Owner clicks "Share" in workflow report header
  → Share link generated, copied to clipboard
  → Owner shares link in Slack, Notion, email, or Confluence
  → Viewer opens the link
  → Viewer reads the SOP
  → Viewer sees "Made with Ledgerium" CTA
  → Viewer clicks CTA → lands on signup page
  → Viewer creates free account
  → Viewer installs extension
  → Viewer records their first workflow
  → Viewer shares their SOP
  → Loop repeats
```

Each completed loop produces one new user who has already seen a real SOP output. These users arrive with demonstrated intent and an aha moment already experienced. Their activation rate should exceed the cold-signup baseline materially — the comparable benchmark from Loom's shared video model is approximately 2-3x higher activation rate for viewers who sign up from a shared link versus cold organic signups.

### Viral coefficient model

Viral coefficient K = (share rate) × (viewer-to-signup conversion rate)

Inputs:
- Share rate target: 25% of activated users share at least one SOP within 7 days (from GROWTH_STRATEGY.md secondary activation metric)
- Average viewers per shared link: estimate 3-8 (Slack channels, Notion pages, email threads)
- Viewer-to-signup conversion: target 8-15% (comparable to Scribe's shared guide conversion; Notion's public page signup rate is approximately 5-10%)

Conservative scenario (K = 0.25 × 4 × 0.08 = 0.08):
At K = 0.08, organic sharing contributes meaningful but not self-sustaining growth. Each 100 new users generates 8 additional signups from sharing alone. At this stage, the loop amplifies paid and direct acquisition rather than replacing it.

Target scenario (K = 0.25 × 6 × 0.12 = 0.18):
At K = 0.18, sharing is a substantial secondary acquisition channel. For every 100 direct signups, 22 additional users arrive through the share loop within 30 days.

Aspirational scenario (K = 0.40 × 8 × 0.15 = 0.48):
Approaching self-sustaining at K > 0.5. Requires share rate above 40%, which means the share prompt must become a natural part of the post-recording flow, not an optional afterthought.

**The lever with the highest sensitivity is share rate, not viewer conversion.** The design effort should prioritize making the share action obvious and well-timed over optimizing the landing page conversion.

## 3. Attribution and CTA Design

### "Made with Ledgerium" placement

There are three placement zones. All three must be present for the shared view to function as an acquisition surface.

**Zone 1 — Sticky top banner**

A slim bar (40px) pinned to the top of the shared view viewport. Always visible. Does not overlap content.

```
[Ledgerium wordmark]  This SOP was generated automatically from a live browser recording.
[CTA button: "Record your own workflows free →"]
```

This banner is always present on shared views from Free and Pro accounts. Enterprise accounts with custom branding can replace the Ledgerium wordmark with their own logo, but the CTA language remains.

**Zone 2 — Inline attribution at SOP footer**

At the bottom of the SOP content, before the process map section:

```
Generated by Ledgerium AI
Every step in this SOP was captured from a real browser recording — not written from memory.
[Record your own SOP free — no credit card required]
```

This placement catches viewers who scroll through the entire SOP. By this point they have consumed the product's full value and are most receptive to the pitch.

**Zone 3 — Process map caption**

Below the process map visualization:

```
This process map was generated automatically from the same recording.
See what Ledgerium generates for your workflows — [Start free]
```

### CTA copy options to test

Test these in sequence during Phase 2 of the rollout (A/B test using the share token as the split key):

| Variant | Headline | CTA |
|---------|----------|-----|
| A (control) | "Made with Ledgerium AI" | "Try it free" |
| B | "This SOP was generated from a live recording — not written by hand." | "Record your first workflow free" |
| C | "Your SOPs could look like this — generated automatically in minutes." | "Start free. No credit card." |
| D | "Stop writing SOPs from memory." | "See what Ledgerium generates from your workflows →" |

Variant D speaks most directly to the pain identified in ICP_DEFINITION.md. The positioning decision (Option A: Process Documentation Leader) supports this framing. Test B and D against the control first.

### Signup page from shared link

When a viewer clicks any CTA on the shared view, they land on a dedicated signup page, not the generic homepage. The URL carries a `?ref=shared_sop` parameter so acquisition source is captured.

The signup page headline reflects what they just saw:

```
"You just saw what Ledgerium can generate.
Now record your own workflows — free.

No credit card. No screenshots.
Your first SOP in under 10 minutes."
```

Below the headline: a minimal signup form (email, password). The sample workflow referenced in GROWTH_STRATEGY.md loads immediately on first login so the viewer-turned-user sees another SOP output within 60 seconds of creating their account.

## 4. Sharing Distribution Channels

Shared links will propagate across different surfaces. Each requires slightly different behavior.

### Slack

Slack unfurls URLs with Open Graph metadata. The shared SOP page must include:

```html
<meta property="og:title" content="[Workflow Name] — SOP generated by Ledgerium AI" />
<meta property="og:description" content="[Step count] steps · [Duration] · [System names] · Generated from a live browser recording." />
<meta property="og:image" content="[generated OG image URL]" />
<meta property="og:type" content="website" />
```

The OG image should be a dynamically generated card showing the workflow name, step count, and the first 3 step titles in the SOP's visual style. This requires a server-side image generation endpoint (Satori or equivalent). Without a visual OG card, Slack unfurls show a blank preview and click-through rates drop significantly (Loom reports approximately 40% higher click-through for cards with visual previews versus text-only unfurls).

**Behavior in Slack context:** The OG preview shows in the Slack thread. A colleague sees "Create Purchase Order — SOP generated by Ledgerium AI / 12 steps · 4m 30s · SAP, Outlook" and clicks through to the full SOP.

### Notion

Notion embeds Ledgerium shared links as bookmark cards using OG data. No special integration is required. The same OG metadata that works for Slack works for Notion. The shared SOP URL pasted into a Notion page becomes a bookmark block with the workflow name, description, and preview image.

This is the most important distribution surface. Operations team leads (the primary ICP) work in Notion. A Ledgerium SOP link embedded in a Notion process documentation page reaches other ops team leads in the same org without any additional effort from the owner.

### Confluence

Confluence's smart links also use OG metadata. Same implementation as Notion. Confluence is the dominant documentation tool in enterprise and financial services — both high-priority verticals from ICP_DEFINITION.md. Shared SOPs appearing as smart links in Confluence pages reach compliance and operations team members in regulated industries.

### Email

No special handling needed for email — the URL is a plain link. The CTA in the shared view footer performs the acquisition work. Consider adding a "Copy link" button alongside the primary share button in the workflow report header so users can paste the link directly into email without an extra step.

### Direct link (copy and paste)

The share action should copy the link to clipboard immediately on click with a "Copied!" confirmation. No intermediate modal. The UX_FLOWS.md spec already defines this behavior: button shows "Shared" with a green tint and "Copy link" appears as a secondary button. This is correct — do not add friction.

### LinkedIn and Twitter/X

Some users will share SOP links publicly on social media, particularly ops professionals who want to demonstrate their documentation quality. The OG card design should work for these surfaces. Do not build dedicated social sharing buttons — they add visual clutter for a behavior that will happen organically if the output quality justifies it.

## 5. SEO and Discoverability

### Should shared SOPs be indexed?

Default: No. Shared SOP pages are set to `noindex` by default.

Rationale: Most SOPs contain internal process information that owners do not intend to be publicly discoverable. Default indexing would be a trust breach with enterprise and compliance-sensitive users. An ops team lead who shares a purchase order SOP with their team does not expect Google to index it.

Override: Owners can explicitly opt in to indexing from the workflow settings. This is a Pro feature.

### Public SOP Gallery

A public gallery of opt-in shared SOPs is a Phase 2 SEO play, not a Phase 1 launch requirement. Do not build it for initial launch. The value case:

- SEO benefit: pages titled "How to create a purchase order in SAP — SOP" or "Salesforce opportunity management — process documentation" would rank for long-tail search queries from ops professionals
- Social proof: a gallery of real SOPs demonstrates product quality to prospective users better than marketing copy
- Network effect: browsing the gallery converts casual visitors with no prior referral

Prerequisite before building: at least 50 opt-in shared SOPs with real content, representing diverse workflows and systems. This will not exist at initial launch.

When building the gallery (Phase 2), structure each gallery entry as a static page with proper schema markup for `HowTo` structured data. Google indexes HowTo content for rich results in search. An SOP for "how to submit an expense report in Workday" with proper HowTo schema would generate organic impressions from the exact query intent that matches Ledgerium's ICP.

### Canonical URL handling

Each shared SOP has one canonical URL (`/shared/[share_token]`). If the owner revokes and re-shares, the old token is deactivated and a new token is issued. Any bookmarked old URLs return a graceful "This SOP is no longer publicly shared" page — not a 404 — with a "Create your own SOP with Ledgerium" CTA.

## 6. Analytics and Measurement

### New events required

These events are additions to the existing event taxonomy in SUCCESS_METRICS.md.

| Event | Trigger | Properties |
|-------|---------|------------|
| `sop_shared` | Owner clicks Share and share link is generated | workflow_id, user_id, share_token, plan_tier |
| `shared_sop_viewed` | Any page load of a shared SOP URL | share_token, workflow_id, referrer, viewer_session_id (anonymous) |
| `shared_sop_cta_clicked` | Viewer clicks any CTA on shared view | share_token, cta_zone (banner / footer / process_map), cta_variant |
| `shared_sop_signup_completed` | New user signs up with `?ref=shared_sop` in URL | share_token, referral_workflow_id |
| `shared_sop_viewer_activated` | A user who arrived via shared link completes first real SOP view | user_id, source_share_token |
| `share_link_revoked` | Owner deactivates a share link | workflow_id, user_id, share_token |

### Funnel to measure

```
Shared SOP views (unique)
  → CTA clicks                           [viewer engagement rate]
  → Signups with ?ref=shared_sop         [viewer-to-signup conversion]
  → Extension installs from shared ref   [signup-to-install conversion]
  → First real SOP viewed                [install-to-activation rate]
  → Second workflow recorded in 7 days   [activation-to-retention rate]
  → Pro upgrade within 30 days           [activation-to-revenue rate]
```

Each stage of this funnel is a separate measurement. The first two stages (view to click, click to signup) are measurable immediately. The later stages require the PostHog activation funnel already defined in SUCCESS_METRICS.md, augmented with a `source=shared_sop` cohort filter.

### KPIs that define success

| KPI | Definition | Target | When to measure |
|-----|-----------|--------|-----------------|
| Share rate | % of activated users who share at least one SOP within 7 days | > 25% | Weekly, from Phase 1 launch |
| Viewer-to-signup conversion | % of unique shared SOP page views that result in a signup | > 8% | Weekly, from Phase 1 launch |
| Viral K coefficient | (share rate) × (avg views/share) × (viewer-to-signup rate) | > 0.15 | Monthly |
| Shared-referral activation rate | % of shared-ref signups who complete first real SOP view | > 40% (target exceeds cold-signup baseline of 30% because viewers arrive with prior product exposure) | Monthly |
| Shared-referral 30-day Pro conversion | % of shared-ref activated users who upgrade to Pro within 30 days | > 7% | Monthly, from Phase 2 |

### What to watch in the first 30 days

1. Which CTA zone (banner, footer, process map) generates the most clicks. This determines where to invest copy optimization effort.
2. Which referrer sources drive shared SOP views (direct, Slack, Notion, email). This reveals where the ICP is naturally sharing.
3. Whether the OG card is generating previews in Slack (check this manually in the first week by posting a test shared link in a Slack channel).
4. The median number of views per shared link. If most links get 0-1 views, the owner is sharing privately (1:1 email) rather than to channels. If median is 5+, the link is reaching teams.

## 7. Tier Gating Strategy

The gating strategy must balance two competing needs: maximize the viral reach of shared SOPs (which argues for making shared views as feature-rich as possible) against preserving upgrade incentive (which argues for gating premium share features behind Pro).

The resolution: the shared view content is never gated. Every viewer sees the full SOP and process map regardless of the owner's plan. What is gated is the owner's control over and visibility into the sharing experience.

### Feature matrix

| Feature | Free | Pro ($149/mo) | Enterprise (custom) |
|---------|------|---------------|---------------------|
| Generate share link | Yes | Yes | Yes |
| Revoke share link | Yes | Yes | Yes |
| Full SOP visible to viewers | Yes | Yes | Yes |
| Process map visible to viewers | Yes | Yes | Yes |
| View count (owner dashboard) | No — owner cannot see how many times SOP was viewed | Yes | Yes |
| Password protection | No | Yes | Yes |
| Custom expiry (link auto-expires after N days) | No | Yes | Yes |
| Remove "Made with Ledgerium" banner | No | No | Yes |
| Custom domain for share URL | No | No | Yes |
| Agent intelligence visible to viewers | No (gated to authenticated Pro users) | No (gated to authenticated Pro users) | Custom |
| Bulk share (share all workflows in a portfolio) | No | Yes | Yes |
| Share analytics export (CSV of view data) | No | Yes | Yes |

Rationale for Pro gating of view count: knowing how many colleagues opened your SOP is a meaningful value signal, not a vanity feature. It gives the owner evidence that their documentation is being used. This is also an engagement hook that brings the owner back to the dashboard — which is a retention driver.

Rationale for keeping "Made with Ledgerium" on Pro but not Enterprise: Enterprise customers negotiate custom contracts and often require white-labeling for internal tools. Pro users are still individual contributors for whom the attribution is a reasonable trade-off for the value they receive. Do not offer attribution removal to Pro users — this is a meaningful Enterprise differentiator.

Rationale for never gating the shared SOP content: the viral loop depends on viewers seeing the full product value. A shared SOP that shows "upgrade to see the full SOP" is actively harmful to acquisition — it makes the owner look bad in front of their colleagues and destroys trust in the product before a prospective user signs up. Scribe does not gate its shared guide content. Notion does not gate its public pages. The content must be free to view.

## 8. Launch Sequence

This plan respects the phased approach in GROWTH_STRATEGY.md and the current Phase 1 / beta-validation context. The feature is launched in three phases corresponding to product maturity and validated user volume.

### Phase 0 — Prerequisites (complete before any share feature ships)

These items must be confirmed before Phase 1 begins:

- [ ] Extension auto-sync is functional (PRD_v2.md P1-3). The share flow is meaningless if most recordings cannot produce a viewable web app workflow. Do not launch sharing before auto-sync works.
- [ ] Onboarding wizard is complete (PRD_v2.md P0-1). The signed-up viewer must have a working path to first SOP. If the post-signup experience breaks, shared link acquisition converts to nothing.
- [ ] PostHog events are confirmed firing for `signup_completed`, `first_sop_viewed`, and `workflow_uploaded`. Without instrumentation the viral funnel is not measurable.
- [ ] The WorkflowShare schema is confirmed in the database. If it exists as schema-only (per PRD_v2.md), confirm it is migrated and the endpoint exists or is built.

Estimated engineering effort for Phase 0 dependencies (already defined in PRD_v2.md): 2-3 weeks for auto-sync + onboarding wizard in parallel.

### Phase 1 — MVP Share Feature (build target: 1-2 weeks after Phase 0)

Scope:
- Share button in workflow report header (already designed in UX_FLOWS.md)
- Share token generation (POST /api/workflows/:id/share)
- Public shared SOP view route (/shared/[token]) rendering SOP + process map
- "Made with Ledgerium" sticky banner (Zone 1) + footer CTA (Zone 2)
- Revoke share (DELETE /api/workflows/:id/share)
- OG metadata tags for Slack/Notion unfurl
- Analytics events: `sop_shared`, `shared_sop_viewed`, `shared_sop_cta_clicked`, `shared_sop_signup_completed`
- Signup page variant for `?ref=shared_sop` traffic
- Share prompt in post-recording onboarding flow: after first SOP is viewed, prompt: "Would you share this SOP with a colleague?" with one-click share

Not in Phase 1 (defer to Phase 2):
- OG image generation (deliver text-only OG first; add visual card in Phase 2)
- View count on owner dashboard
- Password protection
- CTA A/B testing infrastructure

**Phase 1 launch milestone:** 10 shared SOP links created by real beta users. Measure: first 30 days of share activity. If fewer than 10 shares in 30 days with 25+ active users, the prompt design or the SOP quality is insufficient — diagnose before Phase 2.

### Phase 2 — Conversion Optimization (build target: 4-6 weeks after Phase 1)

Trigger to proceed: Phase 1 milestone met (10+ shares, measurable viewer-to-signup data).

Scope:
- OG image generation for Slack/Notion visual unfurl cards
- View count displayed on owner workflow detail page (Pro-only)
- CTA A/B test infrastructure (split by share_token mod 2 or similar)
- Process map CTA (Zone 3)
- Share analytics: which workflows are shared most, referrer breakdown per share token
- Upgrade prompt triggered when Free user attempts to view their share analytics or enable password protection
- "Copy link" button visible alongside Share button without requiring re-click

**Phase 2 milestone:** Viewer-to-signup conversion rate above 8%. Measured over 60 days of Phase 2 data.

### Phase 3 — SEO and Scale (build target: 3-4 months after Phase 1)

Trigger to proceed: Viral K coefficient above 0.10 for two consecutive months. At least 50 unique shared SOPs with 3+ views each.

Scope:
- Opt-in indexing toggle in workflow settings (Pro feature)
- Public SOP gallery with HowTo schema markup
- Password protection (Pro feature)
- Custom link expiry (Pro feature)
- Bulk share / portfolio share (Pro feature)
- Canonical URL redirect handling for revoked tokens
- Share analytics CSV export (Pro feature)

**Phase 3 milestone:** At least 500 organic impressions per month from indexed SOP pages. At least 5 signups attributed to public gallery or indexed SOP discovery within 60 days of launch.

## 9. Risk Mitigation

### Risk 1: Content leaks — owner shares sensitive workflow data publicly

**Description:** An ops team lead shares a workflow that contains process steps referencing sensitive internal systems, field names, or business logic. The shared SOP is publicly accessible without authentication.

**Mitigations:**
- The SOP output never exposes field values — only field names and step descriptions (enforced by the existing SOP standard: "Sensitive fields: Named but values never exposed" per process-map-sop-standard.md). This structural constraint limits the damage.
- Share links are non-guessable (UUID v4) — they are not discoverable by enumeration.
- Owners can revoke links immediately.
- The shared view is `noindex` by default — search engines will not crawl and cache the content.
- Consider a "sensitive content" warning during share: "Before sharing, confirm this SOP does not contain confidential process details. Viewers do not need to log in to see this content." One-click dismiss. Logs a `share_warning_acknowledged` event.

**Residual risk:** Low. The product architecture (no screenshots, no field values, no video) inherently limits the information density of a shared SOP compared to a screen recording or screenshot guide.

### Risk 2: Abuse — automated creation of spam SOPs shared publicly for SEO

**Description:** Bad actors create Ledgerium accounts, generate SOPs, share them publicly, and opt into indexing to create spam pages that abuse Ledgerium's domain authority.

**Mitigation:**
- Opt-in indexing is a Pro feature behind a $149/month paywall. This is a meaningful abuse barrier.
- Rate limit share link generation per user (max 50 active share links for Free, 500 for Pro).
- Review queue for opt-in indexed pages before they are submitted to sitemaps (manual review is feasible at Phase 3 scale).
- Noindex default prevents Phase 1 and Phase 2 shares from being indexed regardless.

**Residual risk:** Low in Phase 1 and Phase 2. Monitor in Phase 3 when indexing is introduced.

### Risk 3: Viewer experience is underwhelming and generates no signups

**Description:** Viewers open the shared SOP, skim it, and do not click the CTA because the output quality or the CTA placement does not convert.

**Mitigations:**
- Phase 1 deploys with CTA in two positions (sticky banner + footer) before A/B testing infrastructure exists. This is a deliberate over-placement to generate early click data.
- The signup page copy is specifically written for referred viewers, not generic marketing copy (see Section 3).
- The post-signup experience (sample workflow + onboarding wizard) must deliver a second SOP impression within 5 minutes of signup. If it does not, viewer activation rates will not exceed cold-signup rates.
- Set a 30-day kill switch: if viewer-to-signup conversion is below 3% after 30 days of measurable data, pull the feature back into design review rather than continuing to optimize copy on a broken funnel.

**Residual risk:** Medium. This is the most likely failure mode. Mitigated by treating Phase 1 as a measurement phase rather than a growth phase.

### Risk 4: Owners do not share

**Description:** The product generates share links but no one uses them, making the viral loop theoretical rather than real.

**Mitigation:**
- The share prompt must be embedded in the post-recording flow, not buried in a settings menu. After first SOP view: "Would you share this with your team?" with one-click share. This is the highest-leverage placement.
- Frame sharing as "sharing your documentation with your team" not "sharing a Ledgerium link." The motivation is the owner's professional credibility, not product promotion.
- Consider an email reminder 48 hours after first SOP view: "Your workflow is documented. Here's how to share it with your team." Triggered only if the user has not yet shared.

**Residual risk:** Medium. Share rate is the highest-sensitivity variable in the viral coefficient model. If it stays below 15%, revisit the share prompt design and the overall SOP output quality before attributing the problem to the CTA.

### Risk 5: Competitor replication

**Description:** Scribe or Tango sees the shareable SOP feature and builds an equivalent in weeks, neutralizing the distribution advantage.

**Assessment:** Scribe already has shared guides. Tango has shared flows. Ledgerium's shareable SOP is not a novel concept in the category — it is Ledgerium catching up to table stakes while adding the process map as a visual differentiator.

**Mitigations:**
- The moat is not the share link mechanism. The moat is the SOP quality and the process map — Scribe cannot produce a process map, and Tango was acquired by Datadog and has shifted focus. The share link is a distribution mechanism for a genuinely differentiated output.
- Speed to launch matters: ship Phase 1 before open beta so that the earliest adopters establish the sharing behavior.

### Risk 6: Shared view page performance under load

**Description:** A shared SOP goes viral in a Slack channel, generating hundreds of simultaneous page loads that exceed the single-instance SQLite + local filesystem deployment (flagged as a constraint in PRD_v2.md).

**Mitigation:**
- At current scale (Phase 1-2), this is not a material risk. A single viral Slack post is unlikely to generate more than 50-100 concurrent loads.
- Shared SOP views are read-only and can be cached at the CDN level. The shared view route should be cacheable by share token (cache-control: public, max-age=300 with ETag invalidation on revoke).
- The PRD_v2.md identifies local filesystem and SQLite as constraints requiring migration before Phase 4. Monitor for traffic spikes and escalate to engineering if a shared SOP generates more than 500 views in a 24-hour period.

## 10. Revenue Impact Model

### Assumptions

All assumptions are grounded in available data. Adjust as beta data materializes.

| Input | Value | Source |
|-------|-------|--------|
| Active free users (Month 3) | 50 | GROWTH_STRATEGY.md Month 3 target |
| Share rate (target) | 25% | GROWTH_STRATEGY.md secondary activation metric |
| Average views per shared link | 5 | Conservative estimate; Loom reports median 4-8 views per shared video |
| Viewer-to-signup conversion | 10% | Mid-range of Scribe/Notion comparable |
| Shared-referral activation rate | 40% | Assumes pre-exposure advantage over cold signup (30% baseline) |
| Shared-referral 30-day Pro conversion | 7% | Slightly above cold-signup Pro conversion target of 5% |
| Pro plan price | $149/month | Current pricing |

### Month 3 baseline (no sharing)

- 50 active free users
- 30% activation rate = 15 activated users
- 5% Pro conversion = 0.75 Pro users → effectively 1 Pro user
- MRR from sharing: $0

### Month 3 with sharing active

- 50 free users × 25% share rate = 12.5 shares created
- 12.5 shares × 5 average views = 62.5 total shared SOP views
- 62.5 views × 10% viewer conversion = 6.25 new signups from sharing
- 6.25 signups × 40% activation = 2.5 additional activated users
- 2.5 activated × 7% Pro conversion = 0.175 Pro users from sharing

At this scale, sharing contributes less than 1 additional Pro user per month. This is expected — sharing is a compounding mechanism that requires scale to produce meaningful revenue. The month 3 value is not revenue; it is the beginning of the loop.

### Month 6 with sharing compounding

Assumptions: 150 total active free users, share rate maintained at 25%, loop has been running for 3 months.

- 150 free users × 25% share = 37.5 active shares
- 37.5 × 5 avg views = 187.5 views/month
- 187.5 × 10% conversion = 18.75 new signups/month from sharing
- 18.75 × 40% activation = 7.5 activated users/month from sharing
- 7.5 × 7% Pro conversion in 30 days = 0.525 Pro conversions/month from sharing
- Monthly incremental MRR from sharing: 0.525 × $149 = $78.23

Still modest in absolute terms but the share loop is now responsible for approximately 12% of total new signups, which compounds further into Month 9 and beyond.

### Month 12 projection (illustrative)

At 500 active free users with sharing established as a habit:

- 500 × 25% = 125 active shares outstanding
- 125 × 5 = 625 shared SOP views/month
- 625 × 10% = 62.5 new signups/month from sharing
- 62.5 × 40% × 7% = 1.75 Pro conversions/month from sharing loop alone
- Monthly incremental MRR: 1.75 × $149 = $260.75

By Month 12, the share loop contributes a modest but self-sustaining revenue stream without additional acquisition spend. More importantly, the 62 signups per month from shared SOPs reduce the effective cost per acquisition across all other channels.

### Revenue upside levers

The model above uses conservative assumptions. The following would materially change the output:

1. **Viewer-to-signup conversion reaches 15%:** Month 12 MRR from sharing grows to approximately $390/month.
2. **Share rate reaches 40%:** Month 12 MRR from sharing grows to approximately $420/month.
3. **Average views per share reaches 10** (occurs when users share to large Slack channels rather than 1:1): Month 12 MRR doubles.
4. **Pro conversion for shared-ref cohort reaches 10%** (possible if the onboarding wizard successfully delivers the second aha moment): Month 12 MRR triples.

The compounding scenario (all four levers together) at Month 12 produces approximately $1,500/month in MRR attributable to the share loop — representing a meaningful percentage of total MRR at projected scale.

### Non-revenue value

Separate from direct revenue, shared SOP links produce:

- Brand impressions in Slack channels, Notion pages, and Confluence docs where the ICP works — organic marketing that cannot be purchased
- Social proof artifacts that can be referenced in sales conversations and content ("Ledgerium SOPs are being shared in 50+ enterprise Slack channels")
- A feedback loop for SOP quality — if a shared SOP generates zero views, it is a signal that the workflow is either private by nature or the output quality did not warrant sharing

## Handoff Summary

**Launch objective:** Deploy a shareable SOP link feature that turns every workflow output into an acquisition touchpoint, with measurable attribution from viewer to signup to activation.

**Target audience for the feature:** Existing activated users (ops team leads) who have recorded at least one workflow and seen its SOP output. The share prompt targets this cohort.

**Core message to the viewer:** "This SOP was generated automatically from a real browser recording — not written from memory. Record your own workflows free."

**Activation motion from shared link:** Viewer sees SOP → clicks CTA → lands on `?ref=shared_sop` signup page → sees second SOP (sample workflow) within 5 minutes of signup → installs extension → records first workflow → shares their own SOP.

**Measurable experiments to run in Phase 2:**
- CTA copy variants (A/B on `sop_shared_cta_clicked` rate by variant)
- Share prompt placement (post-recording vs. post-SOP-view: which generates higher share rate)
- OG card vs. text-only unfurl (measure click-through from Slack posts)

**Phase 1 success gate:** 10+ shares created by real users, viewer-to-signup conversion measurable (any data), no critical bugs on the shared view route.
