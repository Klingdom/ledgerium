## Agenda Item 2 — Ceiling-rule discipline under Mode 5 directed precedence

### Finding
Mode 5 directed precedence is draining the pool-size ceiling rule's enforcement teeth on long sequences. MR-004 Change A's companion-burn-down rule (guardrail 8) is a one-shot obligation discharged by iter 019 — the remaining 5 Path B iterations had zero burn-down pressure. Pool accumulation during Path B went 21 (post-iter-019) → 24 (020) → 33 (021) → 30 (after 022 close-sweep) → 29 (023) → 33 (024), a net +12 against a rule that is supposed to force burn-down when pool > 8. The rule's **threshold (>8) is aspirational; operational floor is currently ~30**.

### Evidence
- `IMPROVEMENT_BACKLOG.md` portfolio summary: "Pool 29 → 33 open (ceiling deeply violated)" — phrase "deeply violated" has appeared in 4 consecutive iteration entries (021, 022, 023, 024).
- `ITERATION_LOG.md` iter 021 Governance Signals: "Mode 5 operating-mode precedence bypasses ceiling for iter 022 (per clause 7 exclusion). Companion-burn-down obligation discharged at iter 019."
- Iter 022 *did* close 4 follow-ups opportunistically (#48/#49/#50/#52) — but those closures happened because the frontend-engineer absorbed them into the a11y/polish scope, not because the ceiling rule forced them. The rule was inoperative; the closures were a happy accident of PRD structure.
- MR-004 Change A rule text: "at least one iteration of the sequence MUST be a `burn-down` selection" (singular). For a 6-item sequence, one burn-down equals 17% of the sequence; for a 2-item sequence, 50%. The rule does not scale.

### Verdict
**Scale the companion-burn-down rule with sequence length AND add a hard-stop ceiling.** Two concrete changes:

1. **Companion-burn-down scaling:** for a Mode 5 sequence of N ≥ 3 items with pool > 8, require ⌈N/3⌉ burn-down iterations distributed within or preceding the sequence. A 3-item sequence = 1 burn-down (existing behavior preserved); a 6-item sequence = 2 burn-downs; a 9-item sequence = 3 burn-downs. The spacing is unconstrained (all at start, all at end, or distributed) so the user retains scheduling flexibility, but the aggregate debt-carrying cost of a long directed sequence is bounded.

2. **Hard-stop ceiling at pool > 15:** when pool size crosses 15 open follow-ups AND a Mode 5 directed sequence is in progress, the coordinator MUST halt the next directed item and schedule a burn-down iteration regardless of user directive. User override requires a logged `hard-ceiling-override: user-ack` with a written rationale — and override cannot be used more than once per sequence (second breach is a mandatory stop). Justification: the operational data says a >8 ceiling is ignorable under Mode 5; a >15 hard-stop provides the backstop the soft ceiling promised but didn't deliver.

## Agenda Item 3 — Density-trigger accounting across Path B

### Finding
The density trigger fired 4 of 6 counting iterations (020/021/022/024) and resolved as `acknowledged, carried forward` in all 4. The rationale paragraphs for those 4 responses share a common shape: "the follow-ups are scope-boundary / post-launch / PRD-commitment artifacts, not iter-scope re-scope candidates." That pattern — while honest — indicates the rule's taxonomy is under-fitted to the PRD-build iteration type. The three-option menu (`re-scoped` / `root-cause-analyst invoked` / `acknowledged, carried forward`) has no explicit slot for "single-outcome iteration surfaced legitimate adjacent scope that guardrail 7 correctly rejected" — which is what 020/021/022/024 actually did.

Detailed-PRD iterations (020 metrics + 021 UI) systematically produced more follow-ups than bug-fix iterations (023 zero) or governance iterations (018 zero). Density scales with contract complexity, not with iteration quality. The rule as written treats density as an alarm; for PRD-build iterations it is more often a scope-guard output.

### Evidence
- `ITERATION_LOG.md` iter 021 L419: "UI-build iterations against detailed PRDs naturally surface downstream integration/polish/infrastructure work; these are scope-guard outcomes, not scope violations."
- `ITERATION_LOG.md` iter 024 L87–93: 4 follow-ups all tagged with explicit scope-boundary reasoning (#58 deferred growth-strategist review; #59 blocked on #51 analytics; #60 architecture decision; #61 post-launch calibration). None is an iter-024 re-scope candidate.
- `ITERATION_LOG.md` iter 022 L300: "3 follow-ups is AT the clause-3 density threshold (3+), not exceeding it. Of the three, #55 is a trivial tooling fix, #56 is a governance-only decision (no code), and #57 is a scheduled 14-day-out event (not a defect)."
- Bug-fix counter-example iter 023 L184: "None generated." Same iteration had 1-outcome scope (BUG-07 fix), no PRD adjacency to surface.

### Verdict
**Add a fourth density-response option** for the pattern observed 4× in Path B. This is the smallest change that aligns the rule to the actual data while preserving the alarm function for the other three cases. Proposed wording:

`density-response: scope-guard-adjacent; N follow-ups are legitimate adjacencies rejected by guardrail 7(b); root cause is PRD surface area, not iteration quality` — requires (a) one-sentence justification per follow-up OR (b) reference to a PRD section that anchors each follow-up as post-launch / architecture / blocked-on-other-item.

This option is stricter than `acknowledged, carried forward` because it demands a per-follow-up anchor rather than a single rationale sentence — the audit cost is higher. So `acknowledged, carried forward` remains available for genuine "we're just deferring" cases, and `scope-guard-adjacent` picks up the PRD-adjacency pattern that's currently collapsing into the catch-all.

Retroactive classification of Path B responses under the new option:
- Iter 020: `scope-guard-adjacent` (v1 retirement = D2 PRD commitment; stale-chip param = API-layer adjacency; sort params = PRD §6 adjacency). All 3 anchored.
- Iter 021: mixed — 8 of 9 are `scope-guard-adjacent` (PRD-assigned or architecture-decision); #53 is `acknowledged, carried forward` (coordinator-generated, retracted scope expansion, not a PRD adjacency).
- Iter 022: `scope-guard-adjacent` (#55 tooling gap surfaced during triage; #56 governance-only; #57 PRD D1 post-launch commitment). All 3 anchored.
- Iter 024: `scope-guard-adjacent` (#58 deferred specialist review; #59 blocked on #51; #60 architecture decision; #61 post-launch calibration). All 4 anchored.

Zero Path B responses would have been `acknowledged, carried forward` under the new option — the catch-all was misfiring.

But there is no evidence `ux-designer` was consulted on the a11y wiring (kebab focus management, aria-live regions, error role="alert") — those are UX-on-accessibility decisions, not pure engineering.
- **Iter 024:** chip copy rewrite (§4.1b) rewrote 5 user-visible strings. `ITERATION_LOG.md` L38 explicitly records: "`growth-strategist` review is deferred as a polish follow-up (scope-expansion protocol — would have expanded to multi-outcome bundle)." ### Verdict
**Add an agent-invocation gate to the Delegation Rubric**: when an iteration's described scope changes user-visible copy (≥3 strings) OR exposes a new UX surface (component ≥1 or new aria pattern ≥1), the relevant specialist (`growth-strategist` for copy, `ux-designer` for UX surface) MUST be invoked as adjacent — NOT deferred as a follow-up. The adjacency is evidence-based review, not engineering handoff; it can be a lightweight consult (≤30 min) and does not need to scope-expand the primary iteration. This closes the "deferred as follow-up" bypass pattern that iter 024 L38 documents.

Secondary verdict: `system-architect` should be invoked as primary or adjacent at the start of any iteration that implements a new contract ≥ 200 LOC of pure module surface (iter 020 was 305 LOC of pure module — threshold would have fired). Current Delegation Rubric triggers architect only on "architectural decisions"; shifting to LOC-threshold on pure-module surface catches the iter-020 retro.

## Audit-Intake Pattern (MR-005 Change D-5 / MR-004 Change D)

Audit-style artifacts (e.g., `PRICING_AUDIT_001.md`, security audits, accessibility audits, code-health audits) produce large debt surfaces that would overwhelm the live backlog if promoted wholesale and would break the pool-size ceiling rule's diagnostic value.

2. **P0-only live promotion at intake.** Only P0 items enter the live backlog at intake time. All other severity classes remain in the audit artifact as cold pool until a promotion trigger fires.
3. **Cold-pool row tagging.** Any cold-pool item that is later promoted to live must carry `Birth iter` = either the audit-producing iteration number OR `audit-intake` (a canonical anchor meaning "promoted from an audit cold pool"), whichever is most specific.
4. **Promotion path 1 — P0 burn-down creates a slot.** When a live-backlog P0 row derived from the audit closes, the next-highest cold-pool item MAY promote to live backlog (coordinator discretion, or triggered by area-relevance).
5. **Promotion path 2 — PRD-trigger promotion.** Any item in the audit cold pool MAY be promoted to the live backlog if and only if a newly-approved PRD explicitly cites it as a hard blocker in a numbered Dependencies section. The promoted row's `Birth iter` field is `PRD-promoted` with the promoting-PRD path noted. No promotion on implicit or narrative dependency — citation must be enumerated and reader-verifiable.
6. **No other promotion paths.** Coordinator judgment or "we should probably look at this" is NOT a valid promotion path — it breaks the pool-size ceiling rule's protection.

Rationale: the pattern was validated by `PRICING_AUDIT_001.md` (iter M3@016→17) where 4 P0 items entered live, ~27 P1/P2/P3 held cold, and one cold-pool item (BUG-07) promoted cleanly via PRD trigger (iter 018, closed iter 023).
